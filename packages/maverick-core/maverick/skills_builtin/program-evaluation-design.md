# What this skill does

Designs a defensible evaluation for a program or intervention: a logic model linking inputs to outcomes, evaluation questions, an appropriate design (e.g., pre/post, comparison group, RCT where feasible), and a measurement plan. Produces an evaluation design document a program team and an evaluator can execute and a funder can review.

# Steps

1. Pull the program's stated goals, target population, activities, and any funder requirements from `knowledge_search` (program docs, grant agreements, prior reports). Do not invent outcomes the program never claimed.
2. Build the logic model: Inputs -> Activities -> Outputs -> Short/Intermediate/Long-term Outcomes. Mark each assumed causal link and note external factors that could confound it.
3. Frame 3-6 evaluation questions tied to the outcomes, then select a design proportionate to stakes and feasibility (note threats to validity for each option, and why a comparison/control group is or isn't viable).
4. Define measures for each outcome: indicator, data source, instrument, collection cadence, and baseline. Flag where no measure exists yet.
5. Report the design (logic model, questions, design choice, measurement matrix), stating assumptions, data gaps, and the rigor/feasibility tradeoffs you made. Hand off to the program team and evaluator for a methods decision.

# Notes

Output is wrong if the logic model claims impacts the program isn't designed to cause, if the design is over-engineered for the decision it informs, or if measures lack a real data source. Cite the program documents each goal and indicator came from; mark anything inferred as unverified. This is a draft design — choosing a study design (especially randomization or anything affecting participant treatment) and committing budget are human/IRB decisions. Do not use for pure financial audits or for monitoring dashboards that need no causal claim.
