# What this skill does

Designs a Target Operating Model (TOM) that translates a strategy or transformation objective into how an organization will actually run. Produces a coherent TOM spanning people, process, technology, and governance, with the current-to-target gaps named and a sequencing view for closing them.

# Steps

1. Pull the anchoring inputs with `knowledge_search`: the stated strategy/ambition, scope (which business unit/function), and any current-state artifacts (org charts, process maps, system inventory). Record what is documented vs. what you are inferring.
2. Define the design principles (e.g. centralize vs. federate, build vs. buy, customer-segment alignment) and confirm they trace to the strategy. List 4-7; if any are assumed rather than sourced, mark them.
3. Detail each TOM layer: People (roles, org structure, spans/layers, capability gaps), Process (core value-chain processes and ownership), Technology (platforms, data, integration), Governance (decision rights, RACI, KPIs, controls). Map current state to target per layer and name each gap.
4. Produce the TOM artifact (layer-by-layer table plus a one-page model diagram described in text), a prioritized gap list, and an implementation sequencing view. Report assumptions, unsourced inferences, and open design decisions; recommend the model — do not enact org or system changes.

# Notes

The output is wrong if design principles don't trace to the strategy, if a layer is described in isolation (a TOM is internally consistent or it is not), or if current-state claims are fabricated rather than sourced from `knowledge_search`. A TOM is a recommendation: org restructuring, role elimination, and platform decommissioning are irreversible and stay with named human owners and a steering committee. Do not use for narrow process-improvement asks (use a process-redesign skill) or where scope is undefined — pin scope first.
