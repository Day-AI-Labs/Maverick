# What this skill does

Builds a competency model for a role family: a named set of competencies, each with leveled definitions and observable behaviors per level. Output is a calibrated framework usable for hiring, leveling, and development, grounded in the family's existing roles and JDs rather than a generic catalog.

# Steps

1. Define the scope: which role family, which levels (e.g. IC1-IC6 or M1-M3), and the intended use (hiring vs promotion vs development) — these change what behaviors must be observable. knowledge_search for any existing competency framework or leveling guide and reuse its vocabulary so the model is consistent.
2. knowledge_search the JDs and rubrics across the family; cluster recurring expectations into 5-9 distinct competencies (mix of technical/functional and behavioral/leadership). Avoid overlap — each competency should be independently assessable.
3. For each competency, write a one-line definition and then leveled descriptors: concrete, observable behaviors that escalate by scope, autonomy, and impact across the chosen levels. Anchor the highest and lowest levels first, then fill the middle.
4. Hand off the model as a draft with a coverage note (which existing roles it was derived from), and flag assumptions — inferred level boundaries, competencies added without source evidence, and that adoption/calibration is a people-team decision.

# Notes

The model is wrong if levels aren't behaviorally distinguishable (descriptors that only swap adjectives like "good" → "excellent"), if competencies overlap so an assessor can't tell them apart, or if a competency was invented with no grounding in real role data. Cite the JDs/rubrics clustered; mark any competency or level boundary that is inferred. Do not present a draft model as adopted policy — calibration across managers is a human step. Not for assessing a specific individual (use the evaluation skill) or for a single role (use the JD skill).
