# What this skill does

Turns a loosely-stated client ask into a structured scoping document that can seed a statement of work. It produces objectives, workstreams, deliverables, timeline, assumptions, dependencies, and out-of-scope boundaries — the spine of an engagement before pricing. Output is a draft a lead consultant reviews, not a signed contract.

# Steps

1. Extract the client's stated objective and success criteria from the brief or intake notes. If success is undefined, list the candidate outcomes as open questions rather than inventing targets.
2. Use knowledge_search to pull comparable past engagements, standard workstream templates, and known delivery patterns for this domain; ground the structure in what the firm has actually done, citing the source artifacts.
3. Decompose the work into 3-6 workstreams, each with concrete deliverables, an owner role, and a rough duration. Capture assumptions (client provides data access, SME availability), dependencies, and an explicit out-of-scope list to prevent scope creep.
4. Report the scoping doc with objectives, workstreams/deliverables, timeline sketch, assumptions, risks, and out-of-scope items. Flag every assumption that materially affects effort and note that pricing and the binding SOW are decided by the engagement lead.

# Notes

The output is wrong if the out-of-scope section is empty (unbounded scope) or if deliverables are activities ("analyze data") rather than artifacts ("data quality assessment report"). Do not fabricate timelines or rates — those require staffing and commercial input. Cite the reference engagements you drew from; mark any pattern you inferred as unverified. This is a draft for human review and client negotiation; never represent it as a committed scope. Skip for tiny T&M tasks where a lightweight ticket suffices.
