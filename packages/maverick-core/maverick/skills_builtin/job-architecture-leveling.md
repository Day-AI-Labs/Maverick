# What this skill does

Defines a job architecture: a set of job families, the levels within each, and the leveling criteria that distinguish one level from the next. Output is a leveling framework with per-level role descriptions (scope, impact, autonomy, required competencies) and a progression map showing how someone advances within and across families.

# Steps

1. Gather the inputs and any existing standard via `knowledge_search`: the organization's current titles/roles, comp bands if provided, and reference leveling frameworks or competency models in the knowledge base. Identify the job families to cover (e.g. engineering, sales, operations) from the actual roster, not a generic template.
2. Establish the leveling dimensions used to differentiate levels — typically scope of impact, autonomy/decision authority, complexity, and breadth of influence — and define how each dimension shifts as level increases. Cite the source framework when one is reused; mark invented criteria as proposed.
3. For each family, draft level-by-level descriptors using consistent dimensions so a given level means a comparable thing across families (horizontal alignment). Map current real roles onto the new levels to validate that the framework fits the population, flagging roles that don't slot cleanly.
4. Produce the progression map (criteria to advance one level, and lateral moves between families) and report the full framework. State which families/levels are grounded in existing roles vs. newly proposed, and note any roles that resisted placement.

# Notes

The framework is wrong if level descriptors are inconsistent across families (so "level 4" means seniority in one and a manager in another) or if it's built from a generic template that ignores the org's actual roles. Do not attach specific salary numbers unless comp data was provided — leveling and pay banding are separate exercises and conflating them creates legal/equity exposure. This is a proposed architecture for HR/comp and leadership to ratify; it is not policy until approved, and it must not be used to re-slot real employees or change anyone's title automatically. Not for one-off title questions or for evaluating a single individual.
