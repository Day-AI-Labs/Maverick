# RFC: Lightwork 2.0

**Status:** RFC (open for comment). Targets the next major after the current
governed-platform line. Non-breaking until the migration tooling
(`maverick migrate`) lands and a deprecation window has elapsed.

## Why a 2.0

The 1.x line grew from a local kernel into a governed, multi-tenant agent
platform (see [`FEATURES.md`](../FEATURES.md)). That growth left three seams
worth a major-version cleanup:

1. **Config surface sprawl.** Opt-in features each added a `[section]`; the
   schema is now wide. 2.0 consolidates related knobs and ships a validating
   loader that rejects unknown keys with a hint instead of silently ignoring
   them.
2. **Two execution entry points.** `Dispatcher` (in-process) and
   `QueueDispatcher` (out-of-process) diverged. 2.0 makes the queue path the
   default contract and the in-process path a backend of it, so behaviour is
   identical whether or not a broker is configured.
3. **Tool catalog scale.** The registry is well past 500 tools. 2.0 introduces
   capability-scoped tool *bundles* so a deployment exposes a curated set by
   default and opts into the long tail, shrinking the prompt and the attack
   surface.

## Scope (what changes)

- **Config v2**: a single `maverick.toml` schema with a versioned `schema = 2`
  header, a `maverick config lint` validator, and an automatic 1.x→2.0
  rewriter inside `maverick migrate`.
- **Unified dispatch**: `Dispatcher` becomes an interface; the in-process and
  arq-queue implementations both satisfy it; `install_from_config` selects one
  with no behavioural difference at the call site.
- **Tool bundles**: named sets (`core`, `code`, `web`, `finance`, `safety`, …)
  resolved at registry-build time; a deployment lists the bundles it wants.
- **Stable public API**: the `maverick.embed` surface (start a goal, stream
  episodes, read cost) is frozen and semver-guaranteed for the 2.x line.

## Non-goals

- No second-language port of `maverick-core` (settled — see the Language
  Bindings council decision in [`ROADMAP.md`](../ROADMAP.md)).
- No change to the shield's fail-open posture or the budget/killswitch
  guarantees.
- No removal of the single-tenant/self-hosted path; multi-tenant stays opt-in.

## Migration

`maverick migrate` will: back up `config.toml`, rewrite it to schema 2, report
any dropped/renamed keys, and run `maverick config lint` to confirm the result
loads. A 1.x config keeps working for the whole 2.0 deprecation window with a
one-line warning pointing at `maverick migrate`.

## Backwards-compatibility window

1.x configs and the current `maverick.embed` calls remain supported until at
least one minor release after 2.0.0. Breaking removals are gated on the
deprecation warnings having shipped in a prior release.

## Open questions

- Should tool bundles be expressible per-role (orchestrator gets `core`, coder
  gets `core+code`)? Leaning yes; needs a config shape.
- Does config v2 absorb `~/.maverick/.env` interpolation or keep it separate?
  (Leaning: keep secrets in `.env`, reference via `${VAR}` as today.)
